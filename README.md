# Collably — Influencer Campaign Collaboration Platform

> Built by **Abdullah Khan** & **Zeeshan Ahmad**.
>
> A modular PERN platform where **brands post campaigns**, **influencers apply or get invited**,
> and both sides track **deliverables** and **chat in real-time** until the work is shipped.

---

## Tech stack

| Layer       | Stack                                                                 |
|-------------|-----------------------------------------------------------------------|
| Frontend    | **React 18 + Vite**, **Tailwind CSS** (custom shadcn-style primitives), React Router, Axios, react-hot-toast, lucide-react |
| Backend     | **Node.js + Express** (ES modules), **Prisma ORM**, JWT + bcryptjs, morgan |
| Database    | **PostgreSQL** with type-safe migrations via Prisma                   |
| Real-time   | **Socket.io** (authenticated, room-per-collaboration)                 |
| Auth        | **JWT** + **bcrypt** with role-based access (Admin / Brand / Influencer) |
| Dev tools   | nodemon, concurrently, vite proxy, Prisma Studio                      |

## Core workflow

1. **Brand** registers, completes brand profile, **posts a campaign** (budget, platforms, niches, min followers, deliverables template).
2. **Influencer** registers, fills profile (niches, social handles with followers/engagement).
3. Influencer **browses campaigns** and **applies**, *or* brand **searches influencers** and **invites** them.
4. Brand reviews applicants (pending → shortlisted → accepted/rejected). On **accept**, a `Collaboration` is auto-created and the campaign's deliverables template is seeded as a checklist.
5. Both sides open the collaboration:
   - **Deliverables checklist**: influencer submits proof URLs, brand approves/rejects with feedback.
   - **Real-time chat** (Socket.io) with typing indicators and per-collaboration rooms.
6. Brand marks the collaboration **completed** when done.
7. **Admin** can suspend users, flag/close campaigns, and view platform stats.

Status enums match the spec:
- Campaign: `open · closed · draft · archived · flagged`
- Application: `pending · shortlisted · accepted · rejected · invited · declined · withdrawn`
- Collaboration: `active · completed · cancelled`
- Deliverable: `todo · in_progress · submitted · approved · rejected`

## Setup

### Prerequisites
- Node.js 18+
- PostgreSQL 14+ running locally (or a connection string)

### 1. Install dependencies
```bash
npm run install:all
```

### 2. Configure environment
Copy [server/.env.example](server/.env.example) → `server/.env` and adjust:
```env
PORT=5050
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/influencer_campaign_platform
JWT_SECRET=replace-this-with-a-long-random-string
JWT_EXPIRES_IN=7d
CLIENT_ORIGIN=http://localhost:5173
```

> First, create the database in psql: `CREATE DATABASE influencer_campaign_platform;`

### 3. Create the database schema (Prisma migrations)
```bash
cd server
npx prisma migrate dev --name init
cd ..
```
This generates the SQL migration, applies it to your Postgres database, and generates the Prisma client. **Run this once before seeding.**

You can also browse the data in a GUI with:
```bash
cd server && npx prisma studio
```

### 4. Seed demo data (optional)
```bash
npm run seed
```
Creates:

### Socket.io events

| Event              | Direction | Payload                                                  |
|--------------------|-----------|----------------------------------------------------------|
| `collab:join`      | client →  | `collaborationId` (ack)                                  |
| `collab:leave`     | client →  | `collaborationId`                                        |
| `message:send`     | client →  | `{ collaborationId, body }` (ack returns message)        |
| `message:new`      | client ←  | the populated `Message`                                  |
| `message:notify`   | client ←  | preview + sender (delivered to peer's `user:<id>` room)  |
| `typing`           | both      | `{ collaborationId, isTyping }`                          |

Socket auth: pass JWT via `io({ auth: { token } })`.

## Architectural notes

- **Single Application model** with a `kind` discriminator (`application` | `invitation`) so the brand- and influencer-initiated flows share one feed, while a unique `(campaign, influencer)` index prevents duplicates.
- **Collaboration is a derived entity** — created idempotently when an Application transitions to `accepted`. The campaign's deliverables template is seeded onto the new collaboration in the same step.
- **Permission matrix in the deliverable controller** — influencers can move to `in_progress`/`submitted` and attach proof; brands can move to `approved`/`rejected` and leave feedback. The controller validates the transition against the requester's role.
- **Real-time and REST are dual-write friendly** — REST `POST /messages` also broadcasts on the io instance attached to `app.locals`, so the UI stays consistent whether the message came in over WebSocket or HTTP.
- **JWT lives in `localStorage`** — sent on every request via an Axios interceptor, and also passed in `io.auth.token` for socket authentication.
- **shadcn-style primitives implemented manually** (no shadcn CLI dependency) — `Button`, `Card`, `Input`, `Badge`, `Avatar`, `Modal`, `Skeleton`, `Empty` — all use Tailwind + `cn()` (clsx + tailwind-merge).
- **Dark-mode glassmorphism** with aurora gradients, faint grid background, brand glow shadow.
